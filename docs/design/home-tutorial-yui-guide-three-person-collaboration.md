# 首页教程 Yui 引导三人协作实施方案

## 1. 先用人话说明这份文档在解决什么问题

这份文档不是写给某一个开发者单独看的，而是写给整个三人小组看的。

它要解决的核心问题不是“谁更忙一点”，而是“这三个人怎样分工，才能让 Yui 首页引导项目真的做出来，同时不把仓库改成一团线头”。

当前项目已经有一套可以工作的通用教程系统，也已经有首页、设置弹层、跨页通信和独立插件面板。换句话说，我们不是在一片空地上从零造一栋楼，而是在一栋已经住人的楼里加一层新结构。

因此，这次协作的重点不是“平均分代码量”，而是：

- 把高冲突文件收口给一个主负责人
- 让另外两个人尽量在低冲突模块里并行推进
- 把“接口设计”和“实际写功能”拆开
- 让主负责人能稳定地合并他人的提交，而不是天天花时间解冲突

如果三个人都同时去改首页主入口和教程核心文件，这个项目很容易在第二周就进入“每个人都写了，但谁都不敢合”的状态。

所以本方案采用：

- `1 名主负责人 / 集成人`
- `1 名演出层负责人`
- `1 名首页交互与跨页负责人`

这是一种明显偏向“可集成、可合并、可落地”的三人模式。

## 3. 体验基线、原始剧本与工程映射

### 3.1 先说清楚：这次协作要保护的不是抽象概念，而是体验

这次三人协作不是单纯把“Yui 引导”拆成几个模块去写，而是要把一段非常具体的角色体验真正做出来。

因此，团队在讨论接口、模块、分工之前，必须先对齐一件事：

- 我们到底要让用户在首页经历一段什么样的流程

只有先把体验基线说清楚，后面的“谁负责哪个文件”才不会变成自说自话。

本项目的体验基线，不是凭空脑补，而是直接保留最初设计方案里的原始剧本文本，再把它映射到当前项目真实的代码结构上。

### 3.2 原始体验方案文本

以下文本作为本项目的原始体验基线，后续所有实现、拆分和联调都应尽量围绕它展开。

#### 开场三句

> 1.“想要找我的时候，随时在这里打字或者发语音都能召唤本喵哦！(介绍基础)”
>
> 2.“要说你一直没理我，我可是会主动跑出来咬你的哦～（哈！！）(介绍主动)”
>
> 3.“好啦不说废话了喵——你看到那个可爱的‘猫爪’了吗，准备好了吗？让我借用一下下你的鼠标吧！”

#### 实机接管主流程

> 【开始实机接管】（在专属桌面上执行）

> 动作一：夺取鼠标
>
> 画面：鼠标指针在屏幕上轻微晃动两下，然后开始平滑移动。
>
> Yui语音：“嘿咻！可算逮住你的鼠标了喵～”

> 动作二：打开插件管理
>
> 画面：鼠标在轮盘/菜单中熟练地找到并点击「插件管理」，弹出一个充满科技感的插件列表窗口（画面上划过WebSearch、B站弹幕、米家控制等插件）。鼠标在列表上快速上下滑动展示。
>
> Yui语音：“还没完呢！你快看快看，这里还有超～～多好玩的插件呢！有了它们，我不光能看 B 站弹幕，还能帮你关灯开空调…… 本喵就是无所不能的超级猫猫神！哼哼～”

> 动作三：打开设置
>
> “当然啦，如果你想让本喵多和你聊聊天也不是不行啦，给我多准备点小鱼干吧，嘿嘿，好了不逗你啦，设置都在这个齿轮里。你看，这里可以穿我的新衣服、给我换一个好听的声音……换一个猫娘或是修改记忆？等一下！你在干嘛？该不会是想把我换掉吧？啊啊啊不行！快关掉快关掉！”(稍微看一下设置有哪些东西)

> 动作四：归还控制权
>
> 画面：设置窗口被瞬间点掉关闭，屏幕干干净净回到初始桌面状态；鼠标指针乖乖回到屏幕正中央，先软乎乎地左右晃了两下（像小猫蹭手撒娇），随后轻轻一顿，彻底静止，完完整整把控制权交还给用户。
>
> Yui语音：“好啦好啦，不霸占你的电脑啦～控制权还给你了喵！可不许趁我不注意乱点奇怪的设置哦！之后的日子也请你多多关照了喵～”

#### 对抗机制（特殊流程）

> 如果在上述【实机接管】期间，检测到用户物理移动鼠标：
>
> 1：被动回弹（不计入打断次数）
>
> 条件：单次位移 `>= 10px`、速度 `>= 0.2 px/ms`，且距离上次回弹 `>= 140ms`。
>
> Yui 反应：鼠标会往用户移动的反方向“较劲”。
>
> 这个阶段只做视觉回弹，不播放台词，也不增加打断次数。
>
> 2：轻微抵抗（第 1 次 / 第 2 次有效打断）
>
> 条件：位移 `>= 32px`、速度 `>= 1.8 px/ms`、加速度 `>= 0.09`，并连续命中 `3` 次采样；两次有效打断之间还需满足 `>= 500ms` 节流。
>
> 语音1：“喂！不要拽我啦，还没轮到你的回合呢！”
>
> 语音2：“等一下啦！还没结束呢，不要随便打断我啦！”
>
> 第 3 句轻微抵抗台词已删除。
>
> 3：激烈抢夺（第 3 次有效打断后进入生气流程）
>
> 条件：累计有效打断达到 `3` 次。
>
> 画面：Yui 的头像出现怒气符号，并终止教程流程。
>
> 语音：“人类~~~~！你真的很没礼貌喵！既然你这么想自己操作，那你就自己对着冰冷的屏幕玩去吧！哼！”
>
> 后果：Yui 隐藏

### 3.3 原始剧本与当前代码现实的映射

上面的剧本必须保留原味，但工程实现不能和当前项目现实打架。

因此，以下映射规则需要作为整个三人小组的共同前提：

#### A. “借用鼠标”不等于控制真实系统鼠标

原始剧本里的“夺取鼠标”“较劲”“归还控制权”在体验上保留原意，但在工程上必须落实为：

- 页面内 ghost cursor 演出
- `body.yui-taking-over` 级别的视觉接管
- 可跳过、可中断、可恢复的统一清理路径

不能做的事情：

- 不直接控制系统真实鼠标
- 不调用 Agent 自动化替用户操作
- 不做 OS 级鼠标锁定

这与主设计文档的边界一致，也与当前项目实际可用能力一致。

#### B. “专属桌面”优先解释为专属接管场景

原始剧本中写的是“在专属桌面上执行”。

在当前代码条件下，这一表述应优先解释为：

- 首页当前窗口中的专属教程接管场景
- 或在首页之上叠加的专属演出层

也就是说，第一阶段不把它实现成真正的系统级“虚拟桌面”或“操作系统独占工作区”，而是先做成用户感知上的“Yui 把当前桌面接管了”。

#### C. “插件管理”在当前项目里不是首页设置菜单的真实项

这点必须全员统一认知。

当前真实代码结构是：

- 首页设置弹层里有：角色设置、API 密钥、记忆浏览、创意工坊
- 插件管理真实页面是独立 Vue 应用
- 入口是 `/api/agent/user_plugin/dashboard`
- 实际会跳到 `/ui`

因此原始剧本中的“打开插件管理”要分阶段落地：

- `Phase 1 / Milestone 2`：在首页真实存在的“猫爪 / Agent / OpenClaw”入口上做插件预演层
- `Phase 3-4`：再真正接入 `/api/agent/user_plugin/dashboard` 与 `/ui`

这既保留了“Yui 熟练翻插件列表”的观感，也不违背当前代码现实。

#### D. “打开设置”必须锚定真实设置入口和真实菜单项

原始剧本里的“设置都在这个齿轮里”完全可以保留，而且当前代码里确实有真实齿轮入口与真实菜单项。

因此动作三应该优先映射为：

- 打开首页真实设置弹层
- 浏览真实存在的角色设置、API 密钥、记忆浏览、创意工坊等入口
- 再用 Yui 的情绪演出把“别把我换掉”“别乱改记忆”这一层人格化表达加上去

#### E. 开场三句不是装饰，它们要对应真实功能锚点

原始开场三句应该分别落到真实功能上：

- “打字或者发语音都能召唤本喵” 对应首页现有的文本输入和语音入口
- “我会主动跑出来咬你” 对应主动搭话 / 主动视觉这类真实能力
- “可爱的猫爪” 对应首页现有的 Agent / OpenClaw 入口

这意味着实现时不能只播语音不落锚点，而要做到“台词和真实功能入口是一一对应的”。

### 3.4 剧本与实现映射表

| 原始剧本片段 | 当前项目中的真实锚点或实现方式 | 当前阶段建议 |
|---|---|---|
| 找我时可以打字或发语音 | 首页文本输入区、语音按钮、现有聊天入口 | 首页真实锚点直接实现 |
| 不理我我会主动咬你 | 主动搭话、主动视觉等现有设置项与行为能力 | 先保留台词，再在设置阶段映射到真实开关 |
| 可爱的猫爪 | 现有 `#${p}-btn-agent` 对应的 Agent / OpenClaw 入口 | 首页真实锚点直接实现 |
| 打开插件管理 | Phase 1 用插件预演层，Phase 3-4 再真实接 `/api/agent/user_plugin/dashboard -> /ui` | 分阶段实现 |
| 打开设置看看能换猫娘、改记忆 | 首页真实齿轮与真实设置弹层、角色设置、记忆浏览等入口 | 首页真实锚点直接实现 |
| 夺取鼠标 / 归还控制权 | ghost cursor + body 接管样式 + 统一清理与回退 | 仅做页面级演出 |
| 轻微抵抗 / 激烈抢夺 | interrupt 监控、回弹、怒退、统一 skip 语义 | 首页真实机制实现 |

### 3.5 三人小组的共同目标

在不推翻现有教程系统的前提下，完成 Yui 首页引导的渐进式落地：

1. 先让原始剧本中的开场三句和首页真实锚点对齐
2. 再完成“猫爪 -> 接管 -> 插件预演 -> 设置一瞥 -> 归还控制权”的首页主流程
3. 再做跨页面 handoff
4. 最后再评估 `/ui/` 插件管理页的独立教程桥接

### 3.6 本轮协作不追求的事情

本方案不要求三个人在第一轮内同时完成以下内容：

- 重写整个教程引擎
- 让教程直接调用 Agent 自动化去“帮用户操作”
- 第一阶段就打通 `/ui/` 插件 Vue 面板的完整引导
- 把所有页面教程全部翻修一遍
- 把“专属桌面”做成系统级虚拟桌面能力

换句话说，三人组的重点是“让原始剧本在当前代码边界内真实成立”，不是“一次做成整个宇宙”。

## 4. 三个人分别扮演什么角色

## 4.1 角色总览

| 角色 | 目标 | 工作性质 | 是否负责合并 |
|---|---|---|---|
| 主负责人 / 集成人 | 把整条链路接起来并稳定落地 | 高耦合、强边界、少量关键实现 + 评审 + 合并 | 是 |
| 开发 B：演出层负责人 | 把 Yui 的“会说话、会演、会动”做出来 | 新模块为主，低冲突实现 | 否 |
| 开发 C：首页交互与跨页负责人 | 把菜单、跨页、handoff 和真实页面入口打通 | 中等耦合，跨模块接线 | 否 |

## 4.2 为什么主负责人不能“什么都自己写”

主负责人在这个项目里最值钱的能力不是编码速度，而是：

- 统一接口
- 控制改动顺序
- 把别人的提交稳定合进来
- 在出现耦合问题时做最后裁决

如果主负责人同时承担大量独立功能开发，他会在最需要评审和集成的时候被自己写的业务拖住。

所以这个角色要刻意克制：

- 只亲自持有高冲突核心文件
- 尽量把独立功能模块留给其他两人
- 把更多精力放在接口、节奏、review、联调和合并

## 5. 分工总原则

### 5.1 单文件单 owner

对于高冲突文件，必须明确唯一 owner。其他人若确实要改，必须先同步，再由 owner 合并。

### 5.2 优先新增文件，不优先改核心旧文件

能用新增文件承载的新能力，不要一上来塞进旧的大文件。

这样做的好处是：

- 减少冲突
- 便于回滚
- 便于主负责人理解和合并

### 5.3 所有人都围绕同一份接口工作

在三人并行之前，必须先冻结三类接口：

- 教程步骤数据接口
- 演出层挂接接口
- handoff 数据接口

没有接口冻结，就没有稳定并行。

### 5.4 先首页，后跨页；先真实入口，后视觉代理

首页是第一战场。

插件管理 `/ui/` 不是第一阶段主战场；在 Phase 1 中，如果需要表现“插件能力”，优先锚定首页真实存在的 Agent / OpenClaw 入口，再使用预演层做演出代理。

## 9. 需要先冻结的接口

在三人正式并行之前，主负责人必须先发出一版“接口冻结说明”。

## 9.1 步骤接口

建议以主设计文档中的 `YuiGuideStep` 为基础，冻结至少这些字段：

- `id`
- `page`
- `anchor`
- `tutorial`
- `performance`
- `navigation`
- `interrupts`

说明：

- 这份接口由主负责人定版
- `tutorial` 字段主要服务于教程骨架
- `performance` 字段主要服务于开发 B
- `navigation` 字段主要服务于开发 C
- 若实际落在 `static/yui-guide-steps.js`，该文件默认由主负责人长期持有
- 开发 B 主要在这份契约里补 `performance` 与剧本节奏
- 开发 C 主要在这份契约里补 `anchor / navigation` 与真实入口映射

## 9.2 Director 接口

建议冻结以下最小接口：

- `startPrelude()`
- `enterStep(stepId, context)`
- `leaveStep(stepId)`
- `handleInterrupt(event)`
- `skip(reason?)`
- `abortAsAngryExit(source?)`
- `destroy()`

其中：

- 接口定义由主负责人发布
- 具体实现由开发 B 负责
- `UniversalTutorialManager` 的调用接入由主负责人负责

## 9.3 Handoff 接口

建议以主设计文档中的 token 结构为基础，冻结至少这些字段：

- `token`
- `token_version`
- `flow_id`
- `source_page`
- `target_page`
- `resume_scene`
- `created_at`
- `expires_at`

在真正接后端签名校验前，可以先做前端可用的开发版结构，但字段形状不要反复变化。

这个接口由：

- 主负责人定字段
- 开发 C 落地实现

## 9.4 场景 ID 与剧本段落命名建议

为了避免三个人对“哪一段剧情”“哪一个步骤”“哪一个跨页节点”的叫法各不相同，建议在真正编码前先冻结一组最小场景 ID。

建议第一版至少包含：

| 场景 ID | 对应原始剧本内容 | 主要 owner |
|---|---|---|
| `intro_basic` | “想要找我的时候，随时在这里打字或者发语音都能召唤本喵哦！” | 开发 B 演出，开发 C 锚点 |
| `intro_proactive` | “要说你一直没理我，我可是会主动跑出来咬你的哦～” | 开发 B 演出，开发 C 锚点 |
| `intro_cat_paw` | “你看到那个可爱的‘猫爪’了吗……” | 开发 B 演出，开发 C 锚点 |
| `takeover_capture_cursor` | 动作一：夺取鼠标 | 开发 B |
| `takeover_plugin_preview` | 动作二：打开插件管理 | 开发 B + 开发 C |
| `takeover_settings_peek` | 动作三：打开设置 | 开发 B + 开发 C |
| `takeover_return_control` | 动作四：归还控制权 | 开发 B |
| `interrupt_resist_light` | 轻微抵抗 | 开发 B |
| `interrupt_angry_exit` | 激烈抢夺 / 生气流程 | 开发 B + 主负责人 |
| `handoff_api_key` | 跨页到 `/api_key` | 开发 C |
| `handoff_memory_browser` | 跨页到 `/memory_browser` | 开发 C |
| `handoff_steam_workshop` | 跨页到 `/steam_workshop_manager` | 开发 C |
| `handoff_plugin_dashboard` | 跨页到 `/api/agent/user_plugin/dashboard` / `/ui` | 开发 C |

这组 ID 的目的不是把体验写死，而是让：

- 主负责人在骨架层可以稳定引用
- 开发 B 在演出层可以稳定配置
- 开发 C 在页面跳转和恢复时可以稳定对接

后续如果场景数量增加，也应在这套命名规则下继续扩展，而不是临时发明一套新叫法。

## 10. 每个人按阶段分别做什么

## 10.1 Milestone 1：先把开场三句和首页真实锚点对齐

### 主负责人

- 在 [static/universal-tutorial-manager.js](../../static/universal-tutorial-manager.js) 中补出 Director 挂接点
- 先建立首版共享场景注册表与场景 ID 骨架
- 明确 step change 生命周期回调
- 决定旧教程键和新版本教程键如何并存
- 在 `templates/index.html` 中挂载新脚本和样式
- 把原始剧本中的三句开场白映射到首页已有教程骨架，而不是另起一套独立流程
- 第一阶段只允许 `sceneOrder` 非空的页面启用 Yui Guide 运行时；当前实际只启用首页
- 第一阶段不允许把旧首页教程整套删掉；它仍然作为 driver 状态骨架存在

### 开发 B

- 创建 `yui-guide-director` 主模块
- 实现气泡、语音、表情桥的第一版
- 完成 `startPrelude` 和 `enterStep / leaveStep` 最小闭环
- 向共享场景注册表补齐首页首批 `performance` 配置
- `startPrelude()` 第一阶段只处理没有挂到旧 tutorial step 上的 intro scene；按当前冻结结果，实际只处理 `intro_basic`
- 不要在 `startPrelude()` 里重复播放已经由 `enterStep()` 承接的 `intro_proactive / intro_cat_paw`

需完成的三句演出基线如下：

1. “想要找我的时候，随时在这里打字或者发语音都能召唤本喵哦！(介绍基础)”
2. “要说你一直没理我，我可是会主动跑出来咬你的哦～（哈！！）(介绍主动)”
3. “好啦不说废话了喵——你看到那个可爱的‘猫爪’了吗，准备好了吗？让我借用一下下你的鼠标吧！”

### 开发 C

- 盘点首页真实入口和设置弹层中的真实锚点
- 向共享场景注册表补齐首页首批 `anchor / navigation` 配置
- 确保“打字 / 发语音 / 主动 / 猫爪”都能在首页找到真实可用入口
- 确保菜单项 ID、DOM 结构、打开时机适合教程使用
- 为后续 handoff 准备统一入口打开方式

### 里程碑完成标准

- 首页能在现有教程步骤上挂出 Yui 演出
- 原始开场三句已经和真实锚点对齐
- 用户能听懂 Yui 在介绍什么，而且页面上真的能看到对应入口
- 还不要求 ghost cursor、插件预演层和跨页接力完全上线
- 这一阶段仍然是“Yui 接管旧首页教程骨架”，不是“Yui 已经完全替代旧首页教程状态机”

## 10.2 Milestone 2：把“实机接管”主流程做出来

### 主负责人

- 把 `skip / destroy / complete` 的统一清理路径收好
- 把 `ANGRY_EXIT` 的退出语义和正常 skip 统一到同一终止通道
- review 开发 B 和 C 的接入方式，避免接口漂移
- 审核“专属桌面”在工程上是否始终被解释为首页接管场景，而没有越界成真实鼠标控制

### 开发 B

- 实现 ghost cursor
- 实现轻度打断、回弹、较劲、怒退的演出逻辑
- 增加首页内菜单预演层与相关样式

需完成的主流程演出如下：

1. 动作一：夺取鼠标
2. 动作二：打开插件管理
3. 动作三：打开设置
4. 动作四：归还控制权

需完成的对抗流程如下：

1. 轻微抵抗（移动鼠标次数 < 3）
2. 激烈抢夺（移动鼠标次数 = 3，触发生气流程）

### 开发 C

- 打通首页设置菜单内引导
- 让教程可以稳定打开设置弹层并落到正确菜单项
- 在首页真实“猫爪”入口上挂出插件预演层，而不是伪造一个首页根本没有的插件管理按钮
- 为“请她离开”“打开某个页面”等动作补出可复用包装 API
- 保证动作三中看到的设置内容，来自真实设置弹层中的真实入口和真实菜单项

### 里程碑完成标准

- 首页教程已经从“高亮导览”明显升级为“角色演出导览”
- 原始剧本中的“猫爪 -> 借用鼠标 -> 插件展示 -> 设置一瞥 -> 归还控制权”闭环已经在首页成立
- skip 和打断不会把页面状态搞脏
- 轻微抵抗与激烈抢夺都已具备稳定语义
- 仍然只停留在首页，不强求跨页

## 10.3 Milestone 3：让剧本从首页走向真实跨页

### 主负责人

- 审核 handoff token 结构与生命周期
- 决定前端开发版 handoff 与未来后端校验版 handoff 的兼容策略
- 收口各页恢复入口

### 开发 B

- 补充跨页前后的演出节奏
- 做 opening / waiting / resumed 等场景表现
- 确保跨页恢复后气泡、语音、情绪状态能重新建立
- 保证首页中的“插件很多”“设置里还能换猫娘、改记忆”这些台词，在跨页版本里不会和真实页面表现打架

### 开发 C

- 实现 `YuiGuidePageHandoff`
- 在 [static/app-interpage.js](../../static/app-interpage.js) 中接入 handoff 事件流
- 为后续真实插件面板路由接入准备 `/api/agent/user_plugin/dashboard -> /ui` 的桥接骨架

需先打通的真实页面接力如下：

1. `/api_key`
2. `/memory_browser`
3. `/steam_workshop_manager`

### 里程碑完成标准

- 首页可以打开目标页并在目标页恢复教程
- 返回或完成流程时状态清晰
- 失败时可以安全回退
- 原始剧本里“看设置”和“看插件”的描述，开始从首页预演层走向真实页面接力

## 10.4 Milestone 4：评估并接入插件面板 `/ui/`

### 主负责人

- 决定 `/ui/` 是否进入当前版本范围
- 若进入，先冻结 `/ui/` 教程桥的最小目标
- 防止首页教程逻辑和 Vue 面板逻辑直接缠死

### 开发 B

### 开发 C

- 负责 `/ui/` 轻量 bootstrap
- 负责在 `frontend/plugin-manager` 中补最小教程桥
- 负责与 `plugin/server/routes/frontend.py` 和 `main_routers/agent_router.py` 的入口关系对齐
- 把原始剧本中的“打开插件管理”从首页预演层升级为真正能进入插件面板的教程节点

### 里程碑完成标准

- `/ui/` 被视为独立页面系统接入，而不是首页教程的一块假 DOM
- 首页与 `/ui/` 的接力关系清晰、可回退
- 原始剧本中的插件展示段落拥有真实面板落点，而不再只依赖视觉代理

## 11.2 合并顺序建议

推荐固定顺序：

1. 先由主负责人冻结共享契约和挂接点空壳，例如场景 ID、`static/yui-guide-steps.js` 的字段形状、Director/Handoff 最小接口。
2. 再合开发 B 的新增演出模块。
3. 再合开发 C 的页面入口与 handoff 逻辑。
4. 最后由主负责人在骨架层完成接线和联调。

这样做的原因是：

- 共享契约先定下来，B 和 C 才不会在同一个接口文件里反复打架
- B 的模块大多是新增文件，第二步合入时冲突仍然较低
- C 的逻辑依赖真实入口和既定接口，放在第三步最稳
- 主负责人最后接 `UniversalTutorialManager`，最容易一次性看清全链路

## 11.3 合并节奏建议

主负责人不要等一周后一次性合并。

建议每天至少设置两个合并窗口：

- 中午一次
- 晚上一次

每次合并前，要求 B、C 提交：

- 本次改动摘要
- 影响的文件列表
- 是否改动了共享接口
- 是否需要主负责人补挂接点

## 11.4 代码评审关注点

主负责人 review 时优先看以下问题：

- 有没有越过模块边界去改别人负责的文件
- 有没有直接改共享契约文件却没有同步 owner
- 有没有把演出逻辑直接塞进骨架层
- 有没有把 handoff 逻辑写死在 UI 回调里
- 有没有出现多个地方各自调用 `destroy()` 的情况
- 有没有新增会导致难以合并的大范围散改

## 12. 每个人交付什么，才算完成自己的职责

## 12.1 主负责人完成标志

主负责人不是看“写了多少代码”，而是看是否做到：

- 接口定版过
- 关键骨架接入过
- 两位开发的提交能稳定合并
- 首页到跨页主链路可跑通
- 异常退出和回滚路径可控
- 原始剧本被保留为统一体验基线，而不是在集成过程中被悄悄改散

## 12.2 开发 B 完成标志

开发 B 不是看“做了多少动画”，而是看是否做到：

- 演出层是模块化的
- 演出能被安全启动、跳过、销毁
- 中断、怒退、回弹不是散落逻辑
- 首页演出配置和 step 关系明确
- 原始剧本中的台词、动作节奏和情绪转折，在实现后仍然成立

## 12.3 开发 C 完成标志

开发 C 不是看“打开了多少页面”，而是看是否做到：

- 首页真实入口被稳定抽象
- handoff 能发送、恢复、失败回退
- 设置菜单和目标页之间的关系清晰
- `/ui/` 是否接入教程有可执行路径，而不是停留在口头约定
- 原始剧本中的每一段表达，都能落到真实存在的页面结构或真实存在的预演层上

## 13. 风险点与提前约束

## 13.1 风险：主负责人被功能开发淹没

应对：

- 主负责人只抓核心骨架和合并
- 新功能尽量外包给 B、C 的独占模块

## 13.2 风险：开发 B 直接改教程骨架

应对：

- B 通过接口提需求
- 主负责人负责在骨架层补挂接点

## 13.3 风险：开发 C 为了方便直接把 handoff 写进零散 UI 回调

应对：

- 把 handoff 单独模块化
- 尽量避免把恢复逻辑写死在按钮点击链路里

## 13.4 风险：共享场景注册表被三个人同时改成一锅粥

应对：

- `static/yui-guide-steps.js` 默认由主负责人长期持有
- 开发 B 只主改 `performance` 与台词节奏相关段落
- 开发 C 只主改 `anchor / navigation` 与真实入口映射相关段落
- 场景 ID 和字段形状不允许各自私扩

## 13.5 风险：三个人都去改首页入口

应对：

- `templates/index.html` 只由主负责人长期持有

## 13.6 风险：过早进入 `/ui/` 阶段

应对：

- `/ui/` 只在 `Milestone 4` 再进入
- 前三阶段只保留首页真实入口 + 演出代理

---

## 14. 三人协作时建议遵守的工作约定

### 14.1 每日同步内容

每人同步时只讲四件事：

- 今天准备改什么
- 会碰哪些文件
- 是否涉及共享接口
- 是否需要主负责人先给挂接点

### 14.2 提交要求

每次提交尽量满足：

- 单一主题
- 影响文件集中
- 提交信息能看出它属于哪个模块

不建议：

- 一个提交同时改演出、跨页、教程骨架三类内容

### 14.3 冲突处理要求

若 B 或 C 必须碰到主负责人持有文件：

1. 先同步原因
2. 由主负责人决定是自己补，还是允许对方临时修改
3. 修改后必须尽快合并，不长期悬空

---

## 15. 最终建议

这三个人的最佳协作姿势不是“每个人拿走三分之一页面”，而是：

- 主负责人抓中枢和合并
- 开发 B 负责把演出系统独立出来
- 开发 C 负责把真实页面入口和跨页接力打通

这样做的好处是：

- 与 [home-tutorial-yui-guide-architecture.md](./home-tutorial-yui-guide-architecture.md) 的分层设计一致
- 与当前仓库真实文件结构一致
- 与“三人协作且一人负责合并”的工作方式一致

如果后续进入实际开发阶段，建议在本文件基础上再追加一份更细的执行清单：

- 每个 Milestone 的任务卡拆分
- 每个人第一周的每日目标
- 接口冻结版本记录
- 联调阻塞项清单

这样三个人就不只是“知道自己大概负责什么”，而是能明确知道：

- 今天应该改什么
- 哪些东西不能碰
- 改完之后交给谁合
- 合并之后下一步往哪里走
