# 首页教程 Yui 引导主负责人分阶段任务拆分

## 1. 先用人话说明这份文档在干什么

这份文档不是讲“三个人怎么分工”，而是单独讲“主负责人每个阶段到底该做什么”。

主负责人在这个项目里，最重要的价值不是写最多代码，而是让项目始终沿着同一条路往前走：

- 该冻结的接口先冻结
- 该收口的高冲突文件只由一个人收口
- 开发 B 和开发 C 的工作能稳定并行
- 最后能把两边提交安全合进来，而不是合并时才发现大家做的不是同一个东西

如果说三人协作文档解决的是“怎么分工”，那这份文档解决的是：

- 主负责人什么时候该下场
- 主负责人下场时应该改什么
- 主负责人不该亲自写什么
- 每个阶段结束时，主负责人手里应该拿到什么结果

---

- [home-tutorial-yui-guide-architecture.md](./home-tutorial-yui-guide-architecture.md)

- 本文档只负责把主负责人的任务按阶段拆开，变成可以执行的工作清单

## 3. 主负责人长期不变的职责

不管项目进行到哪个阶段，主负责人都长期负责以下事情：

- 持有高冲突核心文件
- 冻结并发布共享接口
- 维护共享场景注册表与场景 ID
- 把原始剧本和真实代码锚点保持一致
- 审核是否有人越过边界修改不该改的文件
- 决定每一阶段是否进入下一阶段
- 合并开发 B 和开发 C 的提交
- 为最终联调和回退路径兜底

主负责人长期持有的重点文件建议为：

- [static/universal-tutorial-manager.js](../../static/universal-tutorial-manager.js)
- `templates/index.html`
- `static/yui-guide-steps.js`
- 本文档与相关设计文档

主负责人可以在联调时触达，但不应长期主写的文件：

- [static/app-ui.js](../../static/app-ui.js)
- [static/app-buttons.js](../../static/app-buttons.js)

主负责人不应主动承包的内容：

- 气泡组件视觉细节
- 语音队列的实现细节
- ghost cursor 动画本体
- 首页设置菜单整块实现
- handoff 模块的主体实现
- `/ui/` Vue 面板内部桥接细节

一句话说，主负责人负责“路线、接口、收口、合并”，不负责“把所有功能亲手写完”。

## 4. 阶段总览

| 阶段 | 主负责人核心目标 | 阶段结束时主负责人必须拿到的结果 |
|---|---|---|
| 准备阶段 | 先把共享契约和 owner 规则立住 | 大家按同一套场景 ID、接口、文件边界并行 |
| Milestone 1 | 把开场三句接到真实首页锚点上 | 首页能跑通开场三句，且台词和真实入口对应 |
| Milestone 2 | 把首页接管主流程安全落地 | 首页主闭环成立，`skip / angry_exit / cleanup` 统一 |
| Milestone 3 | 把首页流程延伸到真实跨页接力 | handoff 可用，目标页能恢复，失败可回退 |
| Milestone 4 | 决定并收口 `/ui/` 是否接入 | `/ui/` 不是口头方案，而是明确接入或明确延期 |

---

## 5. 准备阶段：正式并行前，主负责人先做什么

这一阶段不是产品里可见的 Milestone，但对主负责人来说必须先做。

### 阶段目标

让开发 B 和开发 C 在真正开工前，就围绕同一份契约工作，而不是各自理解一套“Yui 引导”。

### 主负责人必须完成的事情

- 确认原始剧本在当前代码中的真实映射，不允许出现“剧本写插件管理，代码却落到设置菜单里”的理解偏差
- 冻结首版场景 ID
- 冻结 `YuiGuideStep` 的字段形状
- 明确 `Director` 最小挂接接口
- 明确 `handoff` 最小字段结构
- 明确以下高冲突文件的 owner：[static/universal-tutorial-manager.js](../../static/universal-tutorial-manager.js)、`templates/index.html`、`static/yui-guide-steps.js`
- 约定合并顺序和合并窗口

### 这一阶段主负责人应该产出的东西

- 一版共享场景注册表空壳
- 一版 `YuiGuideStep` 字段说明
- 一版 `Director` 最小接口说明
- 一版 `handoff` 字段说明
- 一版 owner 规则
- 一版合并节奏说明

### 这一阶段主负责人不该做的事

- 提前把演出层细节写完
- 提前把 handoff 主体逻辑写完
- 提前自己去做插件预演层

### 进入下一阶段的判定

只有当下面四件事都成立，才应该进入 `Milestone 1`：

1. 开发 B 知道哪些场景 ID 归自己配 `performance`
2. 开发 C 知道哪些场景 ID 归自己配 `anchor / navigation`
3. 主负责人已经给出骨架挂接点空壳
4. 三个人都知道哪些文件自己不能碰

## 6. Milestone 1：开场三句与真实首页锚点对齐

### 阶段目标

把原始剧本最前面的三句台词，变成“说什么，页面上就真能对应看到什么”。

### 主负责人此阶段的核心职责

- 在 [static/universal-tutorial-manager.js](../../static/universal-tutorial-manager.js) 中补出 step change 到 Director 的挂接点
- 在 `templates/index.html` 中补脚本和样式装配位
- 建立 `static/yui-guide-steps.js` 首版骨架
- 把 `intro_basic`、`intro_proactive`、`intro_cat_paw` 这些场景先收口进同一套场景注册表
- 审核开发 B 的台词和演出配置是否真的对应到这些场景 ID
- 审核开发 C 的真实锚点是否真的落到这些入口：`#text-input-area`、`#${p}-btn-mic`、`#${p}-toggle-proactive-chat`、`#${p}-toggle-proactive-vision`、`#${p}-btn-agent`

### 主负责人这一阶段的具体任务拆分

#### A. 先定骨架

- 给 `UniversalTutorialManager` 增加最小可用的 Director 生命周期回调
- 保证当前旧教程逻辑还能继续跑，不因为新挂接点直接失稳
- 决定新旧教程键如何并存，避免用户旧状态污染新引导

当前主负责人已经确认的落地策略：

- 先不为了 `intro_basic` 单独插入一个新的首页 driver step
- `intro_basic` 先由 `startPrelude()` 结合注册表里的 `#text-input-area` 锚点承接
- 旧首页教程 step 先只补最小 `yuiGuideSceneId` 映射，不提前重排旧教程顺序
- `UniversalTutorialManager` 只负责发出 `prelude-start / step-enter / step-leave / tutorial-end` 生命周期，不提前写死演出实现
- 第一阶段只在 `sceneOrder` 非空的页面启用 Yui Guide 运行时桥接；当前实际只有首页
- `startPrelude()` 不重复处理已经挂到旧首页 step 上的 intro scene；当前实际只负责 `intro_basic`
- 这一阶段的目标是“接管旧首页教程骨架”，不是“立刻把旧首页教程整套移除”

#### B. 再定共享注册表

- 在 `static/yui-guide-steps.js` 中先收口场景 ID 和字段形状
- 不要求此时把所有场景都配满，但最少要把开场三句对应的三个场景立住
- 确保 `tutorial / performance / navigation / interrupts` 四块结构没有歧义

#### C. 审一次“剧本没有跑偏”

- 确保“打字或者发语音都能召唤本喵”没有只剩台词
- 确保“主动跑出来咬你”没有脱离主动搭话 / 主动视觉真实入口
- 确保“可爱的猫爪”真的落到首页现有 `OpenClaw / 猫爪` 入口

### 这一阶段主负责人必须拿到的交付物

- 可挂接 Director 的首页教程骨架
- 首版共享场景注册表
- 新旧教程键策略
- 一轮已过 review 的开场三句配置

### 这一阶段主负责人不要亲自去做的事

- 不亲自写气泡播放器
- 不亲自写语音队列
- 不亲自写真实锚点枚举逻辑

### 阶段完成标准

- 首页可以在现有教程骨架上启动 Yui 的开场演出
- 三句开场白与真实锚点一一对应
- 开发 B 和开发 C 后续可以在这套骨架上继续并行
- 旧首页教程在这一阶段仍可作为 driver 状态骨架存在，但不再需要继续扩写成另一套新剧情

## 7. Milestone 2：首页接管主流程落地

### 阶段目标

让“猫爪 -> 借用鼠标 -> 插件预演 -> 设置一瞥 -> 归还控制权”在首页成立，同时保证任何时候都能安全退出。

### 主负责人此阶段的核心职责

- 收口整个接管期的终止语义
- 保证 `skip`、`destroy`、`complete`、`ANGRY_EXIT` 最终都走同一套清理路径
- 审核“专属桌面”仍然只是首页接管场景，而不是越界成真实系统鼠标控制
- 审核开发 B 的接管演出没有把逻辑塞回骨架层
- 审核开发 C 的“插件管理”实现仍然是首页预演层，而不是伪造一个假的首页一级入口

### 主负责人这一阶段的具体任务拆分

#### A. 收口终止与清理

- 定义 `ANGRY_EXIT` 与普通 `skip` 的关系
- 确保 `destroy()` 只作为统一清理落点，不被多个地方各自乱调
- 确保接管期注入的临时 DOM、样式类、监听器、音频状态都能被统一移除

#### B. 收口共享场景注册表

- 把这些场景纳入主负责人维护的注册表：`takeover_capture_cursor`、`takeover_plugin_preview`、`takeover_settings_peek`、`takeover_return_control`、`interrupt_resist_light`、`interrupt_angry_exit`
- 审核字段变化是否仍在原先冻结的形状内

#### C. 审核工程边界

- 不允许真实系统鼠标控制逻辑混入首页教程
- 不允许 Agent 自动化链路被当成教程执行器
- 不允许插件管理在这一阶段直接把首页结构写成 `/ui/` 的假壳
- 在进入 M2 实装前，先要求开发 C 提供首页交互包装 API，至少包括：
  - `openAgentPanel()`
  - `openSettingsPanel()`
  - `ensureSettingsMenuVisible(menuId)`
  - `closeSettingsPanel()`
- 要求开发 B 在接管演出中优先调用上述包装 API，而不是继续把页面点击链硬写进 `Director`

### 这一阶段主负责人必须拿到的交付物

- 一条统一终止路径
- 一套经过收口的接管期场景配置
- 一版可安全退出的首页主闭环
- 一次明确的“边界没越线”审查结果

### 这一阶段主负责人不要亲自去做的事

- 不亲自写 ghost cursor 动画
- 不亲自写预演层视觉
- 不亲自实现设置菜单具体点开逻辑

### 阶段完成标准

- 首页主闭环已经成立
- 页面不会因为中断或跳过留下脏状态
- 三人对“接管”语义的理解仍然一致

## 8. Milestone 3：从首页走向真实跨页接力

### 阶段目标

让首页上的剧情不再只停留在“看一眼”，而是能够跳到真实目标页，并在目标页继续引导。

### 主负责人此阶段的核心职责

- 冻结 handoff token 结构
- 冻结 handoff 生命周期和回退语义
- 为各目标页定义统一恢复入口
- 审核开发 C 的 handoff 是否仍然是模块化能力，而不是散落在按钮回调里的硬编码
- 审核开发 B 的跨页演出不会和真实页面内容打架

### 主负责人这一阶段的具体任务拆分

#### A. 先定 handoff 规则

- 明确 `flow_id`
- 明确 `source_page / target_page`
- 明确 `resume_scene`
- 明确 `created_at / expires_at`
- 明确开发版 handoff 与未来签名版 handoff 的兼容策略

#### B. 再定恢复入口

- 决定目标页如何判断“这是一次恢复引导，而不是普通进入页面”
- 决定失败时如何安全回退
- 决定是否需要统一的 resume bootstrap

#### C. 最后做跨页联调收口

- 先接 `/api_key`
- 再接 `/memory_browser`
- 再接 `/steam_workshop_manager`
- `/api/agent/user_plugin/dashboard -> /ui` 只先做桥接骨架，不在这一阶段把 Vue 面板教程硬塞进来

### 这一阶段主负责人必须拿到的交付物

- 一版冻结过的 handoff 字段结构
- 一版统一恢复入口约定
- 至少三个真实目标页可恢复的联调结果
- 一版失败回退策略

### 这一阶段主负责人不要亲自去做的事

- 不亲自实现 `YuiGuidePageHandoff` 主体
- 不亲自把每个目标页的 UI 打开逻辑都写一遍

### 阶段完成标准

- 首页能安全打开真实目标页
- 目标页能拿到上下文并恢复教程
- 失败时不会把用户带进半截状态

## 9. Milestone 4：评估并收口 `/ui/` 插件面板接入

### 阶段目标

不是“无论如何都要把 `/ui/` 接进来”，而是由主负责人明确做出一个版本判断：

- 这一版正式接入
- 或这一版继续延期，但桥接方向清楚

### 主负责人此阶段的核心职责

- 判断 `/ui/` 是否进入当前版本范围
- 如果进入，先冻结最小接入目标
- 防止首页教程逻辑和 Vue 面板内部逻辑直接缠死
- 审核开发 C 的 `/ui/` 入口对齐是否真实可靠
- 审核开发 B 的首页预演层是否还能和真实 `/ui/` 保持体验连续

### 主负责人这一阶段的具体任务拆分

#### A. 做版本判断

- 当前版本是否真的需要 `/ui/` 教程桥
- 如果不接，是否只保留首页预演 + 路由桥接骨架
- 如果接，最小目标是“能进入并恢复”，还是“能进入并完整讲解”

#### B. 冻结 `/ui/` 最小接口

- 首页到 `/ui/` 的 handoff 方式
- `/ui/` 侧的 bootstrap 入口
- `/ui/` 结束后的回退或完成语义

#### C. 收口技术边界

- 不允许为了赶进度把首页 DOM 教程实现直接强塞进 Vue 面板
- 不允许首页预演层和真实面板使用两套互相矛盾的入口语义

### 这一阶段主负责人必须拿到的交付物

- 一份明确的 `/ui/` 范围判断
- 一套 `/ui/` 教程桥最小目标说明
- 一次首页与 `/ui/` 接力关系的联调结果

### 阶段完成标准

- `/ui/` 要么被真实接入，要么被明确延期
- 不存在“文档写能接，代码里其实没人知道怎么接”的模糊状态

---

## 10. 主负责人每个阶段都要重复做的事情

不管进行到哪个 Milestone，主负责人每个阶段都要重复做以下动作：

- 看共享契约有没有被悄悄改形状
- 看开发 B 有没有把演出逻辑塞进骨架层
- 看开发 C 有没有把 handoff 写死到 UI 回调里
- 看高冲突文件有没有被多人同时直接改
- 看原始剧本是否还和真实锚点一一对应
- 看当前阶段是否真的已经满足进入下一阶段的条件

---

## 11. 主负责人的阶段性交付节奏建议

建议主负责人按下面的节奏推进，而不是等大家都写完再收尾：

### 每天开始前

- 看前一天是否有共享接口变化
- 看今天谁会碰高冲突文件
- 先把当天需要的挂接点或空壳补出来

### 中午合并窗口

- 优先收开发 B 的新增模块或配置变更
- 快速判断是否影响共享契约

### 晚上合并窗口

- 收开发 C 的真实入口与 handoff 相关改动
- 再做骨架层接线和联调

### 每个阶段结束前

- 出一份“这个阶段已经收口什么、下阶段允许做什么”的简短说明

---

## 12. 什么时候主负责人必须停下来重新对齐

如果出现下面任一情况，主负责人不应该继续硬合，而应该先重新对齐：

1. `static/yui-guide-steps.js` 的字段形状被多人改动
2. 开发 B 和开发 C 使用了不同的场景 ID 叫法
3. 原始剧本的某一段已经找不到真实页面锚点
4. `/ui/` 的接入范围在没有明确决策时被提前实现
5. 多个地方开始各自调用 `destroy()`、各自处理终止语义

---

## 13. 这份文档最终想帮主负责人做到什么

到最后，主负责人不应该只是“知道自己是负责人”，而是应该明确知道：

- 这一阶段自己先改什么
- 这一阶段自己不能替别人写什么
- 这一阶段该先合谁的代码
- 这一阶段结束时必须收口哪些东西
- 什么时候可以进下一个 Milestone

如果做到了这一点，主负责人就不会被项目拖着跑，而是真正掌握整个 Yui 首页引导项目的推进节奏。
